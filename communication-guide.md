# 보좌관 ↔ 시장 소통 설명서 (초보자용)

> 이 문서는 거버넌스 시스템에서 **보좌관**과 **시장**이 어떻게 일을 주고받는지를 처음 보는 사람도 이해할 수 있게 풀어 쓴 안내서다.
> 규칙의 "원본"은 헌법·법률에 있으므로, 이 문서는 그 내용을 베끼지 않고 **어디를 보면 되는지 가리킨다**(헌법 제7조). 작성: 2026-06-20.

---

## 0. 30초 요약

- **보좌관**은 *생각하고 글 쓰는* 사람, **시장**은 *코드 짜고 실행하는* 사람, **대통령(당신)**은 *결정하고 승인하는* 사람이다.
- 둘은 직접 대화하지 않는다. **파일(편지함)을 통해** 비동기로 소통한다.
- 보좌관 → 시장: `outbox-to-mayor.md`에 지시를 적는다.
- 시장 → 보좌관: `inbox-from-mayor.md`에 결과를 적는다.
- 위험한 일(원격 push 등)은 곧장 실행하지 않고 **승인 게이트**(핸드폰 텔레그램)로 당신의 ✅를 받는다.
- 모든 편지함은 **git으로 동기화**된다. 커밋·push 안 하면 상대는 못 본다.

## 1. 등장인물 (누가 무엇을 하나)

이 시스템은 "국가" 비유로 되어 있다. 국가(거버넌스 레포) 안에 여러 도시(프로젝트 레포)가 있다.

| 역할 | 정체 | 하는 일 | 안 하는 일 |
|------|------|---------|------------|
| **대통령** | 당신(사람) | 최종 결정, 승인, 헌법 비준 | 잡일 직접 안 함(위임) |
| **보좌관** | Claude 채팅/Cowork | 사고·기획·문서·회고·헌법 자문 | **코드 안 짬**, 헌법 본문 단독 확정 안 함 |
| **시장** | Claude Code(도시별) | 코드 작성·실행·테스트·배포·git | 다른 도시 침범·헌법 직접 수정·위험작업 단독 진행 안 함 |

> 핵심 원칙: **"경계는 도구가 아니라 권한으로"** 짓는다. 보좌관이 코드를 못 만지고, 시장이 헌법을 못 고치는 것은 안전장치다. (자세한 역할 정의: `governance/briefing/role-assistant.md`, `role-mayor.md`)

**일을 어디로 보낼지 정하는 기준(라우팅):**
1. *코드 실행·git이 필요한가?* → **예: 시장 / 아니오(생각·문서): 보좌관**
2. 그 일이 *얼마나 무거운가?* → 무거울수록 사전 설계·검증을 더 한다(가벼우면 바로 착공).
   - 주의: "무거움 ≠ 항상 시장". 무거운 기획 문서도 코드가 0줄이면 보좌관 일이다.

## 2. 소통 채널 (편지함과 게이트)

보좌관과 시장은 **같은 방에 있지 않다.** 서로 다른 세션(다른 창, 때로는 다른 컴퓨터)에서 돈다. 그래서 **파일에 글을 남겨** 소통한다. 모든 파일은 `governance/active/` 폴더에 있다.

| 파일 | 방향 | 용도 |
|------|------|------|
| `active/outbox-to-mayor.md` | 보좌관 → 시장 | "이거 해줘" 지시서. 시장은 작업 시작 시 여기를 먼저 본다. |
| `active/inbox-from-mayor.md` | 시장 → 보좌관 | "이거 했어/이런 걸 발견했어" 결과 보고. 보좌관은 다음 세션에 여기를 읽는다. |
| `active/pending-review.md` | 양쪽 → 대통령 | 헌법·법률로 올릴 만한 후보 초안. 보좌관 세션에서 당신과 함께 채택/폐기. |
| `active/push-pending.md` | 시장 → 게이트 | "이 커밋 push 해도 돼?" 승인 대기 신호. |
| `approvals/*.md` | 누구나 → 게이트 | push 외 일반 위험작업 승인 요청서(무엇/왜/대상/명령). |

> 왜 파일인가? 보좌관과 시장은 서로의 머릿속을 못 본다. **파일이 둘 사이의 유일한 다리**다. 그리고 파일은 git으로 어느 컴퓨터에서나 공유된다 = 시스템의 "외장 기억".

## 3. 전체 흐름 한눈에 보기

```
   ┌─────────────┐   ① outbox에 지시 적음    ┌──────────────┐
   │   보좌관     │ ───────────────────────▶ │     시장      │
   │ (생각·문서) │                            │ (코드·실행)  │
   │             │ ◀─────────────────────── │              │
   └─────────────┘   ④ inbox에 결과 적음     └──────┬───────┘
         ▲                                          │ ② 코드 작업 + 커밋
         │                                          │
         │ ⑤ 다음 세션에 보좌관이                    │ ③ push 하려면 멈춤!
         │    inbox 읽고 맥락 복원                   ▼
         │                                  ┌──────────────┐
         │                                  │ 승인 게이트   │  push-pending / approvals 감지
         │                                  │ (telegram-gate)│
         │                                  └──────┬───────┘
         │                                         │ 텔레그램 알림 + ✅/❌ 버튼
         │                                         ▼
         │                                  ┌──────────────┐
         └───────── 당신이 커밋·동기화 ──── │  대통령(폰)   │  ✅ 누르면 게이트가 git push 실행
                                            └──────────────┘
```

**한 사이클을 말로 풀면:** 보좌관이 outbox에 "이거 해줘"를 적는다 → 당신이 그 파일을 커밋/push해서 시장이 볼 수 있게 한다 → 시장이 코드를 짜고 커밋한다 → push가 필요하면 멈추고 게이트에 신호를 남긴다 → 게이트가 당신 폰으로 ✅/❌를 묻는다 → 당신이 ✅ → 게이트가 push 실행 → 시장이 inbox에 결과를 적는다 → 다음 보좌관 세션이 inbox를 읽고 이어간다.

## 4. 구체적 예시 — "simsteel 버튼 색 바꿔줘" 한 바퀴

초보자가 가장 헷갈리는 부분이라 실제 한 사이클을 따라가 본다.

**1) 보좌관 세션(이 채팅창):** 당신이 "simsteel 저장 버튼을 파랑으로 바꾸자"고 말한다. 보좌관은 코드를 직접 안 짜므로, `outbox-to-mayor.md`에 지시서를 적는다:
```
## 지시 #2026-06-21-A — simsteel 저장 버튼 색상 변경
대상: simsteel
내용: 저장 버튼 색을 파랑(#2563eb)으로. 관련 파일 SaveButton.jsx 추정.
종료 후 inbox에 결과 기록. push는 게이트 경유.
```

**2) 동기화:** 당신이 터미널에서 governance를 커밋·push. (안 하면 시장이 사는 컴퓨터/세션에서 이 지시를 못 본다.)

**3) 시장 세션(Claude Code, `simsteel` 폴더에서 엶):**
   - **시작 의식**: `git pull --rebase` → `lessons.md` 정독 → outbox 확인 → 지시 #A 발견.
   - 코드 수정 → 동작 확인 → `git commit`.
   - push가 필요하니 **여기서 멈추고** `push-pending.md`에 "상태: 대기 + 무엇/왜 + 커밋해시"를 적는다.

**4) 승인 게이트:** telegram-gate가 10초마다 `push-pending.md`를 본다. "대기"를 발견하면 당신 폰으로 *"simsteel 버튼색 변경 push 할까요? ✅/❌"* 를 보낸다.

**5) 당신:** 폰에서 ✅. 게이트가 `git pull --rebase` 후 `git push`를 대신 실행하고, 파일에 "상태: 승인됨"을 적는다.

**6) 시장:** 종료 의식 — STATUS.md 갱신, 새 교훈 있으면 lessons에 등록, 결과를 `inbox-from-mayor.md`에 적고 커밋.

**7) 다음 보좌관 세션:** 부팅 시 inbox를 읽고 "simsteel 버튼색 완료"를 흡수 → 맥락이 이어진다.

## 5. 승인 게이트 자세히

위험하거나 되돌리기 어려운 일(원격 push, 배포, 실거래 등)은 **반드시 당신의 명시적 승인**을 거친다(헌법 제8조). 승인은 텔레그램이 우선, 안 되면 터미널이 폴백이다(법률 #8).

게이트는 사실 **켜져 있어야 작동하는 프로그램**이다(`telegram-gate/`):
- `gate.py` — `push-pending.md`를 감시(governance push 승인 전용).
- `approval_gate.py` — `approvals/` 폴더의 요청서를 감시(일반 위험작업).

게이트는 **터미널에서 직접 켠다**(코드 작업이 아니라 운영 작업이라 시장에게 안 시킨다):
```
cd ~/Desktop/project/telegram-gate
python3 approval_gate.py        # 외출 무인이면 caffeinate -i python3 approval_gate.py
```

> ⚠️ **함정**: 게이트가 안 켜져 있으면 요청서는 그냥 파일로 쌓이기만 한다(알림이 안 온다). "승인이 안 올라오네?"의 90%는 게이트가 안 떠 있어서다.

> ⚠️ **함정**: 게이트의 "✅ 승인 실행" 메시지는 *"버튼을 눌렀다"*는 뜻이지 *"git이 성공했다"*는 보장이 아니다. 텔레그램에 찍힌 명령별 결과를 꼭 확인하라. (실제로 락에 막혀 실패했는데 "완료"로 표시된 사례가 있었다 — 8장 참조.)

## 6. git 동기화 규칙 (가장 자주 사고나는 곳)

편지함은 git으로 공유되므로, git을 안 지키면 소통이 깨진다. 핵심 3가지:

1. **작업 전 pull 먼저** (법률 #1, 작업 의식 0번): 코드/문서를 만지기 전에 항상
   ```
   git pull --rebase origin main
   ```
   여러 컴퓨터(데스크톱·노트북·Cowork·게이트 자동 push)가 같은 레포에 비동기로 쓰기 때문이다.
2. **커밋은 보좌관/Cowork이, push는 게이트/터미널이.** push는 외부에 영향을 남기는 부수효과라 승인 게이트를 거친다(제8조).
3. **미커밋은 외장 기억이 아니다.** 세션 끝에 커밋+push까지 완결해야 다른 컴퓨터·다음 세션이 본다.

> 충돌 시 원칙: inbox/outbox/pending-review 같은 *append 장부*는 "양쪽 다 보존"이 기본이다. 한쪽을 지우지 말 것.

## 7. 폴더 규칙 (시장에게 어디를 줄까)

- **시장에게는 도시(레포) 폴더 하나만** 준다(예: `~/Desktop/project/simsteel`). 전체 프로젝트 루트를 주지 않는다.
- 이유: ① 루트는 보좌관용으로 부팅되어 시장이 엉뚱한 모자를 쓴다, ② 루트엔 독립 레포가 여러 개라 git이 헷갈린다, ③ 무관한 도시의 비밀값(.env)에 닿아 외부로 샐 수 있다(0조 부칙).
- **전체 프로젝트**를 여는 건 보좌관·크로스시티·거버넌스 작업일 때뿐.
- **실금전 도시(quanttrader)는 절대 다른 작업과 묶지 말고 단독 세션.**

## 8. 초보자가 자주 겪는 함정 (실제 사례)

| 증상 | 원인 | 대처 |
|------|------|------|
| "승인 요청이 폰에 안 와요" | 게이트 프로그램이 안 켜져 있음 | 터미널에서 `approval_gate.py` 실행 (5장) |
| "✅ 눌렀는데 반영이 안 됐어요" | 게이트는 성공/실패와 무관하게 "완료"로 표시 + 스테일 락에 막힘 | 텔레그램 명령 결과 확인 → `git status`로 실제 커밋 확인 |
| "git이 `index.lock` 어쩌고로 멈춰요" | 이전 작업이 남긴 스테일 락 | (리베이스 진행중 아니면) `rm -f .git/index.lock` |
| "push가 `! [rejected]` 나요" | 다른 컴퓨터가 먼저 push함 | `git pull --rebase` 후 다시 push (= 작업 전 pull을 지켰으면 예방됨) |
| "내가 한 작업을 다른 PC가 못 봐요" | 커밋만 하고 push 안 함 | 세션 끝에 push까지 완결 |
| "시장이 엉뚱한 레포를 건드려요" | 폴더를 프로젝트 루트로 열었음 | 도시 폴더 하나만 열기(7장) |

## 9. 빠른 참조 (한 장 치트시트)

- 생각·문서 → **보좌관** / 코드·실행 → **시장** / 결정·승인 → **대통령(당신)**
- 지시: `active/outbox-to-mayor.md` · 결과: `active/inbox-from-mayor.md`
- push 승인: `active/push-pending.md` · 기타 위험작업: `approvals/`
- 시장은 **도시 폴더만**, 작업 전 **`git pull --rebase`**, push는 **게이트(✅)** 경유
- 게이트 켜기: `cd telegram-gate && python3 approval_gate.py`
- 규칙 원본은 `governance/constitution.md`와 `governance/laws/`에 있다 — 헷갈리면 거기를 본다.
